Workflow

  • what is it to me, replacing image-based tools with markup tools
  • can be daunting

Objections

  • takes too long
  • have to do graphics in one program and layout in another
  • stifles creativity, pushes people toward minimal/flat design
  • "aleady swamped" "take longer" "not my job" "stifle creativity" "look certain way, flat"
  • not for everyone, but worked for me, share
  • how did I get here?
  • FW partisan since 1998
  • still use even after discontinuance
  • know all the keyboard shortcuts, pages versus states and all of that
  • ran trainings, debated vs PS
  • traditional waterfall
  • wrote requirement docs and static mockups
  • not worth writing code
  • read, experimented, understood, far from expect
  • could do anything in FW, but not connected to feasibility/desireability

Waterfall Process

  • Separated from developers by department
  • Wrote long requirements documents defining every possible spec
  • developers coded it ad six months later we got to test it
  • we'd find problems then argue over whether they were bugs or change requests
  • QA would say we've already begun regression testing, we can't not launch it
  • very political, inflexible, and quality was not what it should have been

  • Bizonians, Donnie
  • liked me, needed someone who could prototype
  • small team, collaboration
  • nervous, but excited
  • this was harder in before frameworks

frameworks can help

  • Frameworks made easier, don't have to start from scratch
  • like grid system, validation styles, UI elements, forms
  • customize widgets, visual identity
  • frameworks reduce cost of DITB
  • frameworks make learning curve easier
  • transition: benefit 1 - efficiency

kanban process

  • kanban is type of agile, kanban queue sketch, my role in it
  • once set up, realized team very fast
  • especially similar designs, iterations

example of a kanban task

  • we use Asana to track them
  • Backlog, Dev Ready, In Progress, Done, Testing, Production

sketching

  • mainly exploring layouts
  • some visual ideas but not usually detailed
  • do this until I feel comfortable with a concept

go right into Bootstrap

  • use local server version of bizstrap
  • free to iterate on my own as much as I want before committing

quick and dirty at this point

  • code it pretty fast
  • just trying to get my concept up, so use inline styles for now
  • no point in editing my stylesheet until I'm confident of what final will be like

publish to release candidate

  • once I commit I can publish to our release candidate site, send link to others
  • nice to review this way because I do my best proofreading after I click send
  • so I can quickly update little mistakes and no one sees them

code reviews on Jira

  • do another commit to remove inline styles and clean up css
  • keeps them informed on what we're worknig on
  • helps me become a better coder, if I'm doing something in a sloppy way
  • example code review on table pagination-right

Iterate based on feedback

  • customers, developers, PMs all give feedback together
  • nice to uncover all constraints as early as possible without sequencing them

scalable

  • change my colors, all mockups get new colors

scalable

  • metrics example, pain to change all of these manually

speed

  • keyboard faster than mouse, don't have to manually space everything

easy to share

  • can update mockups you've already shared without sending files everywhere

consistency

  • like a pattern library but without all the upkeep
  • best documentation is documentation in practice
  • have to go out of your way to not follow it
  • doing something different will be obvious in code review, so you'll need to justify it

frees developers to develop

  • we hire generalist engineers, they like to solve hard problems
  • most of them don't want to write front-end code (this 1st way you get them on board)
  • better way to communicate with them, they don't want to sit down and read long requirements docs anymore than I want to write them
  • they'd much rather review code and get the idea

better design quality

  • much less disconnect between what I design and what gets built
  • they don't always see how things are supposed to be aligned
  • typography can be unpredictable when going from mockup to browser

easy prototyping

  • very simple to string HTML mockups together to make a clickable prototype
  • easy to prepare usability tests
  • can keep a fully clickable prototype of future features, how they all hang together

includes all the specs

  • can specify how it scales across browser sizes, handle dynamic content, hover states, animated transitions, tab order, content in dropdowns, field placeholders

charts

  • a big thing for me, how do they look with different data?
  • can see how changing inputs or options affects how it will look

keeps design grounded in reality

  • maybe show dribbble mockup

learning curve

  • frustration at first
  • lots of trial and error figuring out why it's not working
  • can't do as much at first, much slower while ramping up
  • temptation to go back to old tools, don't give in
  • well worth the investment in yourself

mastering it feels great

  • like playing with legos

how to do it

  • parter with Devs, get their buy in
  • choose a framework
  • set up a code repository to store commits, do code reviews
  • set up a local server and version you can update on your machine
  • set up a release candidate to store your prototypes that you can link to and update
  • update and add new features over time

resources

  • all kinds of stuff
