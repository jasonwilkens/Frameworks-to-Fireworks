+

  • what is it to me
  • replacing image-based tools with markup tools
  • coding done by someone else, they're making design decisions
  • can be daunting for some

objections

  • takes too long, designing in Photoshop is faster
  • have to do graphics in one program and layout in another
  • stifles creativity, pushes people toward minimal/flat design
  • was for me at first, became fastest
  • especially fast in framework
  • graphically heavy layouts true
  • designing for the web, native medium in touch with constraints
  • how did I get here?

1998

  • FW partisan since 1998
  • still use
  • know it well
  • used in traditional waterfall
  • alongside requirements documents
  • didn't write code, little familiarity
  • could do anything in FW, but not disconnected from feasibility

Bizo

  • came to Bizo, an advertising company
  • product originally for internal use
  • Bizo wanted to use Bootstrap, from Twitter
  • needed designer who could prototype
  • nervous, but excited
  • frameworks made this possible

framework benefits

  • basic, functioning UI out of the box
  • comes with grid system, validation styles, UI elements, forms
  • just add custom widgets, visual identity
  • cost and time to set up lower
  • learning curve for designer much easier

waterfall process

  • touch on development processes
  • came from waterfall
  • siloed from developers
  • design is small and early
  • created reqs and docs and passed them over
  • four or five months to create
  • QA says we can't fix problems because of regression
  • very political, inflexible, and quality was not what it should be

kanban process

  • Bizo figured out how to use kanban
  • type of agile, won't delve deeply
  • it's a queue
  • keep tasks as small as possible
  • work on few tasks at once
  • not as much can go wrong, easy to fix
  • continuous updates

kanban bins

  • Backlog
  • Dev Ready [12]
  • In Progress [10]
  • Test [8]
  • Production [15]
  • simplified view of our bins
  • backlog is unlimited
  • other bins limited
  • tasks pulled along rather than pushed

kanban task

  • we use Asana, a task management app
  • this feature is for ad states
  • pending approval, approved, rejected
  • can see it's in the backlog
  • it's assigned to me
  • requirements minimal

Benefits

  • design quality

    • fewer surprises at release
    • typographic rendering
    • output matches input, less disconnect
    • things like alignment, spacing don't get missed
    • not open to interpretation
    • typography is especially nice

    design quality

    • can specify more in code, easier
    • handle different browser sizes, dynamic content, hover states, animated transitions, tab order, content in dropdowns, field placeholders
    • for this nav, don't need images
    • if something doesn't work right, I know why

    prototyping

    • can build complete prototype
    • future state of all features
    • common baseline for organization
    • usability testing easier

    charts

    • charts are a big deal for Bizo
    • I need to know how they'll look with different data
    • a developer built this library on top of d3
    • in this chart have dashed green line, blue line, area
    • hard/time consuming to draw a chart how the library will
    • by inserting the javascript into the HTML I can see that

    chart markup

    • here's the markup for that chart
    • you can see the data points
    • also define how the scale works, how dates show

    chart styles

    • chart styles go into framework
    • style with CSS like any other element

    keep design grounded in reality

    • here's a very different looking chart from dribbble
    • in fairness probably designed for a very different purpose
    • line has five different gradients, a shadow, glowy piece
    • this is possible to build, but is it worth it?
    • either I or Stephen will need to spend time on that
    • for our org it's better to move on
    • by styling the chart myself, keeps me in touch
    • easy to get divorced from costs like that in photoshop

    scalable

    • all the benefits of css scalability
    • Paul updated Bizo branding guide
    • change my colors, all mockups get new colors

    scalable

    • imagine changing all these progress bars manually
    • even if you're using a feature like symbols in FW, that's more work
    • a couple lines in css

    consistency

    • documentation in practice
    • easy to follow standards
    • engineering team has philosophy that code is it's own documentation
    • should clearly define the proper way of doing something
    • try to carry that over into our framework for design
    • every element should be there, look consistent
    • have to go out of your way to not follow it
    • departures obvious in code review

    engineering

    • let developers develop
    • better collaboration
    • less documentation
    • we have generalist engineers, like to solve hard problems
    • don't want to write front-end code
    • only a few are good at it
    • this is how you get them on board with process
    • better way to communicate with them
    • they don't want to read long requirements, I don't want to write them
    • saves us both time
    • they'd much rather review code and get the idea

    comfort

    • typing faster than dragging
    • keyboard faster than mouse
    • elements get spaced automatically
    • precision, don't have to worry if something is a little off
    • I learned all the keyboard shortcuts for fireworks, but still not enough

    easy to share

    • update on the fly
    • maintain control
    • get rid of outdated designs
    • not emailing images around
    • not exporting and replacing old files
    • code history is always saved
    • don't have old versions floating around
    • send something to client for limited time

    Getting Started

    • initial
    • lots of trial and error figuring out why it's not working
    • can't do as much at first
    • slower while ramping up
    • temptation to go back to old tools
    • well worth the investment in yourself
    • devs were patient with me, they know it saves them time in long run
    • get same feeling of flow
    • joy of creating is just as satisfying
    • even more so, bigger contribution
    • like playing with legos
    • how can you get started?

    how to start

    • partner with engineers
    • choose a framework
    • fork your own version to customize
    • set up repository to track and review code
    • publish release candidate/prototype site
    • set up local server
    • add new features over time
    • parter with devs, get their buy-in
    • organizational support needed
    • choose a framework
    • set up a code repository to store commits, do code reviews
    • set up a local server and version you can update on your machine
    • I use jekyll server, needed help to set it up
    • set up a release candidate to store your prototypes that you can link to and update
    • update and add new features over time
    • one thing we didn't really do well
